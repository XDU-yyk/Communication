[0462] <Heading 2> 第5章 确定性通信与自组网子系统设计
[0463] <Normal> 通信子系统是整个飞行安全控制系统的神经网络。30架无人机在10km作业轴线内协同作业，所有调度指令、状态反馈、感知数据和视频图像都依赖通信链路承载-链路一旦出现不可接受的延迟或中断，集中式协同调度框架就失去了与执行节点的联系，机群退化为各自为政的孤立个体。而应急救援现场的通信条件恰恰是最恶劣的：公网基站可能在灾害中损毁（断网），现场缺乏稳定的电力供应（断电），地形和建筑遮挡造成大量通信盲区。本章围绕如何在三断条件下保证30架异构无人机10km轴线内的确定性通信这一核心命题，从通信需求分析、分层异构自组网架构、控制链路设计、数据链路与图传链路设计、通信可靠性与容错设计五个方面逐一给出子系统级设计方案。
[0464] <Heading 3> 5.1 通信需求分析
[0465] <Normal> 通信需求分析是整个通信子系统设计的起点-如果不先把需要传什么、传多少、多快传到、多可靠这些问题量化清楚，后续的链路设计和技术选型就缺乏依据。30架无人机在救援现场同时作业时，通信子系统承载的业务可以归纳为五大类，每类业务对带宽、时延、可靠性和传输方向的要求差异悬殊。
[0466] <Normal> 先说飞行控制指令，也就是地面平台向空中下发的调度指令。这是通信子系统中优先级最高的一类业务，期望航迹更新、速度与高度调整、紧急消解指令、心跳维持，都归在这里。单条消息极短：航迹路径点序列约50至200字节，速度调整约20字节，紧急指令大约只有10字节。但正是这种轻量容易让人低估它的严苛程度，端到端时延必须控制在50毫秒以内，丢包率必须逼近零。一条丢失的紧急消解指令可能直接导致碰撞，这个逻辑是不可绕过的。控制指令的发送频率与调度周期匹配，为10Hz，单机下行带宽需求约为16kbps，30机合计约480kbps。这一业务必须走独立的、确定性的物理链路，不与任何大流量业务共享，这是一条没有商量余地的前提。
[0467] <Normal> 第二类遥测与状态数据的方向相反，是空中向地面的上行链路。每架无人机持续上载位置、速度、姿态、电量、飞行模式、健康状态与故障码，地面平台维护全局态势、执行冲突检测，依赖的唯一数据来源就是这里。单机遥测数据量约100至200字节每包，发送频率按任务阶段可调：巡航阶段1Hz，任务执行阶段5Hz。取峰值计算，单机上行带宽约8kbps，30机合计约240kbps。时延要求相对宽松，端到端不超过200ms，超过此阈值，地面平台的态势显示会出现可感知的滞后。丢包容忍度低于1%：单包丢失尚可通过插值弥补，不影响态势连续性；连续丢包则会导致跟踪漂移，这是另一条需要标注清楚的边界。
[0468] <Normal> 感知与避障数据是三类中带宽压力最大的一类，同样走上行链路。机载感知系统输出的障碍物检测结果、目标追踪列表和轨迹预测结果，是地面平台更新4D时空占用栅格中环境障碍物占用部分的关键输入。单机感知数据量的波动幅度相当大，开阔场景仅约500字节每帧，复杂城市火场则可达2至5KB每帧。发送频率与LiDAR帧率一致，为10Hz，由此带来的单机上行带宽需求在40至400kbps之间浮动，30机合计峰值可达12Mbps。时延要求最为严格，端到端不超过100ms，超过这一阈值，栅格地图中的障碍物位置已与真实位置产生显著偏差，消解决策将基于过期信息作出。丢包容忍度为5%，单帧检测结果丢失可被下一帧覆盖，检测器本身具有帧间连续性，这为有限程度的丢包提供了缓冲空间。
[0469] <Normal> 第四类是视频与图像数据，方向为上行（空中至地面）。侦察无人机携带的可见光与红外相机，会持续产生高带宽视频流，供指挥员查看火场实况、辅助搜救目标识别。H.265编码1080p@30fps视频流的带宽，通常落在2至4Mbps每路，这个数字会随场景复杂度和编码参数动态浮动，本身并不固定。并发回传路数视在线侦察无人机数量而定，一般在2至4路之间，因此视频上行总带宽需求约为4至16Mbps。时延方面，视频的容忍度相对宽松。指挥员观看的实况画面允许1至3秒的端到端延迟，量级与现场通过公网直播相当，这里真正重要的是画面流畅性，即帧率稳定在25至30fps，而非极低延迟本身。丢包容忍度约为5至10%，单帧丢失表现为轻微卡顿，不影响指挥员对整体场景的判断，此处不必过于苛求。
[0470] <Normal> 第五类是日志与心跳数据，双向传输。心跳消息以1Hz频率发送，每包约20字节，用于维持设备的在线状态；日志数据：飞行日志、消解日志、故障日志在飞行过程中以低优先级于后台传输，速率约为1KB每分钟每机，可容忍数分钟的传输延迟。带宽需求极小，单机低于1kbps。不过这类数据对可靠性有一定要求，不得大面积丢失，原因并不复杂：它是事后故障分析与指标统计的关键依据。

[0474] <Normal> 第一，上下行带宽严重不对称。上行（空中至地面）峰值带宽需求约为16至28Mbps，涵盖感知、视频与遥测；下行（地面至空中）仅约0.5Mbps，承载控制指令与心跳。这个量级的差距，决定了通信子系统在物理层设计上必须以非对称链路为基本前提。
    第二，控制指令与视频数据在时延、带宽、可靠性三个维度上的需求是彼此矛盾的。控制指令要求极低时延、极低带宽、极高可靠；视频数据要求高带宽，同时对时延和丢包有较强容忍度。用同一条物理链路承载这两类业务，必然导致控制系统与视频流竞争信道资源，多旋翼消费级无人机中常见的图传卡顿导致飞控指令延迟故障，根本原因即在于此。因此，控制链路与数据图传链路必须在物理层和链路层实现逻辑隔离，这也正是本系统采用分层异构拓扑加双链路隔离架构的出发点。
    第三，三断环境对通信子系统施加了额外约束，此处需要专门交代。断网意味着不能依赖4G/5G作为备份链路；断电意味着通信设备必须由本系统电源管理子系统独立供电，即200W/3kWh/2.5kWh三级能源架构，并且必须在低功耗状态下维持核心通信功能。这两条约束叠加在一起，对系统的功耗预算与链路冗余设计提出了相当具体的要求。
[0475] <Heading 3> 5.2 分层异构自组网架构
[0476] <Normal> 分层异构自组网是整个通信子系统的拓扑骨架。所谓分层异构有三层含义：分簇-30架无人机不构成一张全连接平网络（那会带来O(N²)的路由开销），而是按空间位置和任务编组划分为若干个簇，每个簇由一个簇头节点管理簇内通信；双链路-同一组网内运行两条逻辑隔离的链路，控制链路承载指令和心跳，数据链路承载遥测、感知和视频，两条链路使用不同的频段和调制方式，互不抢占信道资源；异构-网络中同时存在无人机节点（移动）、地面基站节点（固定或可搬移）和便携式中继节点（可部署），各类型节点的通信能力和角色不同。
[0477] <Heading 4> 5.2.1 簇头选举与簇的建立
[0478] <Normal> 簇头选举，是整个自组网路由体系的逻辑起点。当选的簇头将承担三项核心功能：负责簇内TDMA时隙的分配调度、负责簇间Mesh路由的转发，以及将簇内无人机的状态数据汇总后上报地面基站。
[0479] <Normal> 选举算法采用四维加权评分机制，综合考量四个维度：剩余电量（E）、几何位置中心性（P）、到地面基站的链路质量（L）、当前任务负载（T）。四个维度加权求和，总分最高者当选。剩余电量因子E，权重最高，定为0.35。以剩余电量百分比的归一化值计算。权重之所以最高，原因直接：簇头相较于普通节点，额外承担转发与中继职责，功耗高出约30%至50%，低电量节点根本不具备长期担任簇头的能力。位置中心性因子P，权重0.30。计算方式是该无人机到簇内所有其他成员平均距离的倒数，归一化后参与评分，越靠近簇的几何中心得分越高。背后的工程考虑是：位置居中的簇头到各成员的距离差异小，可以收窄TDMA时隙的保护间隔，这是通信效率上的直接收益。链路质量因子L，权重0.20，以该节点到地面基站的信噪比归一化值计算。簇头不仅是簇内的管理节点，同时也是整个簇与地面基站之间的主通信网关。链路质量差的节点一旦当选，整个簇的对外通信质量都会被拖累，这是系统层面的短板效应，不是个体节点的问题。任务负载因子T，权重最低，定为0.15，以当前已分配但未完成任务数的倒数计算。任务负载轻的节点，计算资源与通信资源的余量更充裕，更有条件承担簇头的额外职责。
[0480] <Normal> 簇头选举的触发时机有三种：初始组网时（任务开始前，所有无人机在地面或低空悬停，按上述评分选出初始簇头，耗时约2-3s完成一轮评分广播和比较）；簇头电量低于阈值（当前簇头剩余电量低于30%时主动发起重新选举，将簇头职责交接给新选出的高分节点）；簇头通信中断（簇头因遮挡或故障失去与地面基站的链路，簇内成员在连续3s收不到簇头心跳后触发重新选举）。重新选举过程中，原簇头在交接完成前仍维持簇内TDMA时隙调度（如果其飞控和链路仍然正常），确保簇内通信不因选举而中断。
[0481] <Normal> 簇的建立过程按以下步骤推进。第一步，邻居发现-每架无人机以1Hz频率在公共控制信道上广播Hello消息（含自身ID、位置、电量、链路SNR），同时监听其他无人机的Hello消息，构建本地邻居表。第二步，簇头候选评分-每架无人机根据邻居表中收集到的信息（通常覆盖通信范围内的全部无人机），计算自身和他人的四维加权评分，选取得分最高的N个节点作为簇头候选（N=ceil(总节点数/8)，即每个簇约5-8架成员）。第三步，簇头确认与成员分配-得分最高的节点声明为簇头并在公共控制信道上广播簇头声明，其他节点收到声明后根据信号强度（RSSI）和跳数选择加入最近的簇头，向该簇头发送Join请求，簇头回复ACK后成员关系确立。第四步，簇间网关建立-各簇头之间通过交换各自簇的成员列表和空域占用范围，建立簇间Mesh路由的初始拓扑。

[0483] <Heading 4> 5.2.2  簇内星型通信
[0484] <Normal> 簇内通信采用星型拓扑-簇头作为中心节点，簇内成员直接与簇头通信，成员之间不直接通信（除非在边缘自治模式下启用机间直连链路做CBBA协商）。星型拓扑在簇内场景中的优势显而易见：单跳直达，延迟确定（成员→簇头仅1跳），时隙分配由簇头集中控制，无信道竞争碰撞。簇内成员数控制在5-8架-超过8架时自动分裂为两个子簇（各选出新簇头），防止单个簇头的TDMA时隙不够分配。
[0485] <Normal> 簇头当选之后，核心职责共四项。其一，TDMA时隙调度：在每30ms的TDMA帧内为簇内成员分配上行发送时隙，保证成员到簇头的上行传输不发生碰撞，具体机制见5.3.1节。其二，状态汇总上报：将簇内所有成员的遥测数据与感知摘要汇聚为单个上报包，统一发送至地面基站，以此压缩上行链路的帧开销，同时降低基站侧的处理负载。其三，指令分发：接收地面基站下发的调度指令，按目标无人机ID转发至对应成员。其四，成员掉线检测：通过心跳轮询持续监测成员在线状态，连续3秒无响应者，标记为离线并即时上报地面基站。
[0486] <Normal> 备份机制的设计逻辑是簇头故障时，引入一个始终在线的备簇头角色。簇选举阶段得分排名第二的节点自动成为备簇头，并在后台持续同步正簇头的TDMA调度表，每个超帧（10帧，300ms）末尾，正簇头向备簇头推送一份调度镜像，约500字节，收到后ACK确认。这一同步机制保证了备簇头手里的调度信息始终是最新的。触发接管的条件是：备簇头连续3s未收到正簇头心跳。一旦触发，备簇头立即广播簇头切换声明，启用最新镜像，并在下一个TDMA帧即以新簇头身份开始调度时隙。整个接管过程不超过300ms，恰好一个超帧周期。通信连续性，由此得到保证。
[0487] <Heading 4> 5.2.3簇间Mesh通信
[0488] <Normal> 簇间通信采用Mesh拓扑，各簇头之间形成全连通的无线Mesh骨干网，依赖多跳路由实现跨簇转发。这一结构主要服务于两个场景。其一是跨簇的冲突消解，当涉事无人机分属不同簇时，两个簇头通过Mesh链路交换占用记录并协商消解方案，这在地面平台不可用的边缘自治场景下尤为重要。其二是链路绕行，某簇头到地面基站的直连链路因遮挡中断时，可经由邻近簇头做Mesh中继，绕道抵达基站。两者合在一起，构成了簇间通信的基本冗余逻辑。
[0489] <Normal> 簇间Mesh路由采用简化AODV（Ad hoc On-Demand Distance Vector）协议-一种按需路由协议，仅在需要发送数据时才发起路由发现，不需要维护全网路由表。简化AODV在标准AODV的基础上做了两项裁剪以适应无人机自组网的移动特征：最大跳数限制为4跳（覆盖10km轴线的簇间通信距离-每跳约2-3km，4跳可达12km，超过10km的作业轴线范围）；路由表的过期时间缩短至2s（标准AODV为30s，缩短是因为无人机位置变化快-以15m/s飞行2s内位移30m，路由下一跳可能已变化）。当簇头A需要向簇头D发送数据但没有到D的路由时，A广播路由请求（RREQ），中间节点B和C逐跳转发RREQ并记录反向路由，D收到RREQ后沿反向路由单播路由回复（RREP）回A，建立双向路由。路由建立延迟通常为50-200ms（取决于跳数和每跳的处理延迟）。
[0490] <Heading 4> 5.2.4 地面基站与中继节点部署
[0491] <Normal> 地面基站是自组网与地面空管平台之间的桥梁。基站包含两个物理上独立的射频模块-控制链路射频模块（1.4GHz频段，TDMA/GFSK调制，接入控制链路自组网）和数据链路射频模块（2.4GHz频段，COFDM/OFDM调制，接入数据链路自组网）。基站配备分集天线-控制链路使用两根全向鞭状天线（增益3-5dBi，间距≥λ/2即约10cm实现空间分集），数据链路使用定向平板天线（增益10-12dBi，指向作业空域中心方向）作为主天线加全向天线作为辅助天线-通过最大比合并（MRC）或选择合并改善链路余量。
[0492] <Normal> 中继节点的部署策略是为解决遮挡场景下的通信盲区。在城市高层建筑火场中，建筑本体对微波信号产生严重遮挡-一座30层钢筋混凝土建筑可以将2.4GHz信号衰减30-40dB，在建筑背面的无人机几乎无法与基站维持链路。中继节点有两种部署方式：预部署-在任务开始前由操作员根据地形和建筑分布估计可能的盲区位置，派遣一架或多架通信中继无人机提前飞抵指定位置悬停（高度选在建筑群平均高度的1.2-1.5倍以上以获取较好的视距条件），在基站和作业无人机之间建立中继链路；动态部署-当系统检测到某簇头或某无人机的链路质量持续恶化（PER超过10%持续10s），地面平台自动规划中继无人机的部署位置并通过任务分配器派遣一架空闲中继机前往补盲。中继节点的核心硬件为一台轻量级射频转发器-接收侧天线接收源信号，经低噪声放大器（LNA）放大后通过数字中频处理转发至发射侧天线，转发延迟（cut-through relay，物理层直通中继）控制在1ms以内，不对系统端到端延迟产生实质性影响。
[0493] <Heading 3> 5.3 控制链路设计
[0494] <Normal> 控制链路承载飞行控制指令、紧急消解指令、空域调整指令和心跳消息，是通信子系统中优先级最高、时延要求最严格的链路。它与数据/图传链路在频段、调制方式和多址接入方式上完全隔离-控制链路运行在1.4GHz频段，采用窄带TDMA加GFSK调制，从物理层保证不被视频流和遥测数据挤占。
[0495] <Heading 4> 5.3.1 TDMA帧结构与时隙分配
[0496] <Normal> 控制链路采用TDMA（时分多址）作为信道接入方式-在一个严格同步的帧周期内，将时间划分为固定数量的时隙，每个时隙分配给一个特定节点发送，从根本上消除CSMA/CA等竞争式接入方式中不可预测的碰撞退避延迟。CSMA/CA在节点增多或高负载时的吞吐量崩溃问题（二进制指数退避导致时延呈指数分布的长尾）对控制指令的确定性要求是不可接受的-控制链路不允许运气不好就需要等待几个退避窗口的不确定性。
[0497] <Normal> TDMA帧的参数设计为：帧周期30ms（控制指令的更新频率为33Hz，高于10Hz的调度频率以保证调度指令有多个下发窗口），每帧划分为36个时隙（30架无人机各1个上行时隙+4个基站下行时隙+2个保护时隙），每时隙持续约750μs（30ms扣除帧头和帧尾保护间隔后除以36），时隙间保护间隔≥50μs（防止因各节点的时钟偏差导致相邻时隙重叠）。每个时隙内可承载一个PHY层数据包-使用GFSK调制（调制指数0.5，带宽-时间积BT=0.5）、符号速率200ksps、256字节最大载荷（含MAC层头部和CRC-16校验），对应空中速率约200kbps。256字节的载荷对控制指令而言绰绰有余-单机控制指令约20-200字节，一条时隙即可完成下发。
[0498] <Normal> 时隙分配的策略如果完全固定的分配是无法适应无人机数量和任务阶段的变化；完全动态的分配则会大幅增加簇头的计算负载和协商开销。半动态策略介于两者之间：初始化阶段，地面平台依据当前在线无人机数量和簇结构生成基础时隙分配表，经簇头下发；运行过程中，簇头按需动态调整，新机入网从预留池分配时隙，离线节点释放时隙归还预留池，紧急冲突场景下可临时占用本帧内的空闲时隙（由簇头在帧头广播中通知）。预留池初始容量为6个时隙，主要用于支持任务进展中陆续加入的后备机。预留池耗尽时，则需触发帧结构重分配，增加时隙数、压缩单个时隙长度，或分裂为两个子簇，三条路径视情况择一。

[0501] <Heading 4> 5.3.2  时钟同步
[0502] <Normal> TDMA的确定性，依赖一个前提：全网时钟严格对齐。偏差一旦超过保护间隔（50μs），相邻时隙的发送便会相互重叠，碰撞随之而来。控制链路的时钟同步采用分层方案。顶层参考时钟来自地面基站的GNSS授时模块，GPS与北斗双模，授时精度±20ns，远优于TDMA所需的μs量级。基站将GNSS绝对时间映射为两个计数器：TDMA帧计数器（每30ms递增一次）和时隙计数器（每帧内取值0至35），作为全网时序的统一基准向下分发。
[0503] <Normal> 簇头通过接收基站下发的周期性信标帧（Beacon，每TDMA超帧=10帧=300ms发送一次）完成与基站的时钟同步。信标帧中携带基站的当前帧计数器和时隙计数器的精确值，簇头接收时在PHY层记录信标帧前导码的到达时刻，结合信标帧中携带的发送时刻，计算自身时钟与基站时钟的偏移并进行补偿。簇内成员通过接收簇头发送的帧头广播完成与簇头的同步-帧头广播在每个TDMA帧的时隙0由簇头发送，携带当前帧的时隙分配表。这一分层同步的级联精度为：基站→簇头同步精度±200ns，簇头→成员同步精度±500ns，成员与基站之间的总时钟偏差不超过±1μs，远低于50μs的保护间隔。
[0504] <Heading 4> 5.3.3 指令优先级机制
[0505] <Normal> 控制链路承载的所有指令被赋予五个优先级级别-不是三个，因为控制链路内部的不同指令之间存在明显的紧迫度差异，三级优先级不足以细化排队策略。P0-紧急消解指令（emergency_stop、紧急避让），优先级最高，直接写入PHY层发送队列的头部，在当前正在发送的时隙结束后立即占用下一个可用下行时隙，不参与常规排队。P1-安全相关指令（return_home、禁飞区闯入警告），优先级次之，在P0之后、所有P2-P4指令之前发送。P2-调度指令（航迹更新、速度/高度调整、悬停/等待），正常调度流程产生的飞行指令，按先进先出排队。P3-空域配置指令（禁飞区更新、通道调整），优先级低于调度指令，因为空域调整通常有10s的响应时间裕度。P4-心跳和其他维护消息，优先级最低，在链路空闲时发送。
[0506] <Normal> 指令队列的发送逻辑为：每个下行时隙开始前，MAC层从指令优先级队列的最高非空级别中取出队首指令，填入时隙载荷。如果高优先级指令在队列中等待超过其延迟预算的50%（例如P0指令等待超过250μs），触发超时告警并在地面平台上提示操作员-但在绝大多数运行场景中，P0-P2指令发送后在下个可用下行时隙（最迟30ms）即被发送，等待时间远低于预算。指令在队列中的最大等待时间由时隙数量和发送速率决定-在30架无人机、每帧4个下行时隙的最坏情况下，一条P2指令最多需等待ceil(已排队的P2和P3指令数/4)帧，通常在2-3帧内即被发送（60-90ms），对于非紧急调度指令这是可接受的。

[0509] <Heading 4> 5.3.4 50ms低时延保障
[0510] <Normal> 50ms，这是控制指令从地面平台应用层发出、到机载飞控完成接收的端到端延迟上限。这个预算不是一个整体目标，而是要在四段链路上逐一分解和落实。地面处理段负责指令的封装、协议栈编码、MAC层排队和PHY层发送，分配2ms。排队段的情况稍微复杂一些，指令在TDMA下行队列中等待时隙分配，实际等待时长取决于当前队列长度和指令优先级：P0指令跳过排队直接调度，P1指令最多等待一个完整的30ms帧周期，因此排队段预算定为30ms（这是最坏情况下的上限，平均等待约15ms）。传输与中继段预算10ms，覆盖从基站天线辐射到机载天线接收的全程，以光速传播，10km距离的空间传播延迟仅约33μs，几乎可以忽略；中继的物理层直通转发延迟不超过1ms，其余主要是PHY层处理和解调的开销。机载接收段预算8ms，完成从天线接收到应用层分发至飞控接口的全流程解封装与协议解码。四段合计恰好50ms。
这个预算的有效性依赖于验证，而不仅仅是设计上的分配。各分段的实际延迟在系统联调阶段通过抓包工具逐段测量，均值、最大值和抖动的统计结果将在验收测试中专门呈现。
[0511] <Normal> 50ms预算在工程上的兑现依赖四项技术措施。第一，控制链路采用简单的GFSK调制加短包格式-调制解调延迟低（GFSK的接收机仅需判决相位变化方向，无需OFDM那样做FFT），短包格式减少了串行化和CRC计算时间。第二，TDMA的确定性调度消除排队延迟的随机长尾-竞争式接入方式中，重传退避可以产生毫秒到百毫秒量级的随机延迟抖动，而TDMA的最坏情况延迟是可精确计算的。第三，中继采用物理层直通转发而非接收→解码→重新编码→转发的存储转发-物理层直通仅放大和重整形信号，延迟在微秒量级。第四，机载LiteOS-M的5ms内核调度周期保证了接收后8ms预算中有足够的时间裕度来完成协议栈处理-最坏情况下协议栈处理需要2-3个内核调度周期即10-15ms来排队获得CPU时间，但通过将协议栈处理任务绑定为最高优先级内核线程（比飞控线程仅低一级），实际的内核调度等待可缩短至1-2ms。
[0512] <Heading 3> 5.4 数据链路与图传链路设计
[0513] <Normal> 数据链路和图传链路，前者承载遥测数据、感知数据和日志数据的上行回传，后者承载视频和图像，两者的共同特点是数据量级远大于控制指令，百KB/s乃至Mbps级，而控制指令不过百B/s，但对单包延迟的敏感度却远低于控制指令。数据链路与图传链路共享同一COFDM物理层（2.4GHz频段），通过上层QoS策略实现差异化服务。它们与控制链路的隔离不在协议层面，而在物理层：控制链路使用独立的射频前端和独立频段（1.4GHz），确保控制指令的传输完全绕开COFDM链路的调度器。隔离点放在物理层而非协议层，意味着控制链路的实时性保障不依赖于上层调度策略的正确执行，这是一种更具防御性的架构思路。
[0514] <Heading 4> 5.4.1  COFDM物理层设计
[0515] <Normal> 数据/图传链路采用COFDM（编码正交频分复用）作为物理层调制方式。OFDM将宽带信道划分为多个正交的窄带子载波，每个子载波独立调制，天然抗频率选择性衰落和窄带干扰。COFDM中的C代表信道编码-在子载波调制之前加入卷积编码（或LDPC编码）和交织，使因某个子载波深度衰落而丢失的比特能够通过编码增益恢复。
[0516] <Normal> COFDM的物理层参数为：子载波总数64个（其中数据子载波48个，导频子载波4个用于信道估计，空子载波12个用于频带保护边带），子载波间隔312.5kHz（64×312.5kHz=20MHz带宽），OFDM符号周期3.2μs加0.8μs保护间隔（CP），合计4μs/符号。每个数据子载波采用自适应调制编码（AMC）-根据接收端反馈的子载波信噪比（SNR），在BPSK(1bit/符号)、QPSK(2bit/符号)、16QAM(4bit/符号)和64QAM(6bit/符号)四档之间动态调整，编码码率在1/2和3/4两档之间调整。四档AMC的净数据速率（扣除导频、CP和编码冗余）分别为：BPSK-1/2约6Mbps，QPSK-3/4约18Mbps，16QAM-3/4约36Mbps，64QAM-3/4约54Mbps。在实际的10km视距或近视距条件下，信噪比通常支持QPSK-3/4或16QAM-1/2档位，链路速率在10-20Mbps之间-足以承载单路视频（2-4Mbps）加全部遥测和感知数据（合计约0.5-12Mbps）。
[0517] <Normal> AMC的链路自适应周期为100ms。每隔100ms，接收端（基站侧）根据该窗口内各子载波的平均SNR计算当前最优档位，通过上行ACK帧反馈给发送端（无人机侧），发送端在下一传输周期切换。对上层而言，这一切换过程是透明的，TCP/UDP流感知不到底层调制方式的变化，感知到的只是可用带宽的升降，以及随之而来的QoS降级或升级。

[0520] <Heading 4> 5.4.2 数据链路设计
[0521] <Normal> 数据链路承载三类上行数据：遥测、感知与日志。说是三类，但它们对时延和带宽的需求实际上差别相当大，遥测数据要求200ms内送达，这是态势显示保持实时性的底线；感知数据要求更严，100ms以内，否则4D栅格中的障碍物信息就已经过期；日志数据则宽松得多，延迟数分钟也无妨。正是这种异质性，决定了数据链路必须在内部建立区分机制。具体来说，是两套并行的手段：紧凑编码，以及三级优先级队列。
[0522] <Normal> 遥测数据的编码在标准JSON或XML格式下，每个浮点值连同键名需要8到15字节；30架无人机、每架20个浮点状态字段、10Hz刷新率，用JSON编码的带宽需求约为1Mbps，对于COFDM链路低端工作点（6Mbps）而言，这个比例并不小。紧凑编码的思路是将所有遥测字段组织为一个定长二进制结构体，每个字段的位宽在满足精度要求的前提下做最小化：经纬度用int32（精度约1cm，无需float64），高度用int16（精度0.1m，范围±3276m），速度用int16（精度0.01m/s，范围±327m/s），电量用uint8，故障码用uint16（支持65536个枚举值）。这样一个结构体约60字节，加上8字节包头与2字节CRC，合计70字节，30机10Hz的总带宽仅约168kbps，大约是JSON版本的六分之一。这一处理在应用层完成，对下层协议栈透明，不引入额外的耦合。
[0523] <Normal> 感知数据的传输逻辑是增量加全量的混合模式。每10帧中，9帧为增量帧，仅传输相对于上一帧的新增目标、消失目标与属性变化，平均体积约为全量帧的20%到30%；第10帧为全量帧，传输完整的检测列表和追踪状态，用于修正增量传输中可能积累的漂移。上述容错逻辑依赖丢包率维持在5%以内，在这个条件下，连续9帧增量中即使丢2帧，第10帧的全量刷新仍能将状态拉回正轨。
[0524] <Normal> 三级优先级队列处理的是链路拥塞时的资源分配问题。遥测数据优先级最高（P_HIGH），感知数据次之（P_NORMAL），日志数据最低（P_LOW）。当发送队列长度超过阈值，意味着排队延迟将突破时延预算，降级按顺序展开：首先丢弃日志数据包，其次将感知数据发送频率从10Hz降至5Hz乃至2Hz，最后在仍然拥塞的情况下将遥测频率从5Hz降至1Hz。但遥测数据包本身永远不会被丢弃。这三步构成了数据链路的内部降级策略，其核心逻辑是：链路质量劣化时，最关键的遥测信息必须保住，其余的按重要性依次牺牲。
[0525] <Heading 4> 5.4.3 图传链路设计
[0526] <Normal> 图传链路承载两类数据：H.265（HEVC）编码的连续视频流，以及单帧高分辨率快照图像。视频侧的关键参数如下：分辨率1080p（1920×1080），帧率30fps，关键帧间隔1秒（每30帧设一个I帧，其余为P帧和B帧），目标码率2Mbps（VBR模式下实际波动区间为1.5至4Mbps），并开启低延迟编码模式，禁用B帧重排序，以压缩编码缓冲带来的延迟。选用H.265而非H.264：在同等画质条件下，H.265的码率开销约为H.264的一半，换言之，2Mbps的H.265可以达到H.264约4Mbps的画质水平。对于频谱资源本就有限的无人机图传场景，这一优势具有决定性意义。
[0527] <Normal> 目标快照模式是图传链路的一个特殊功能-当指挥员需要查看某特定目标的高清细节时（如一栋建筑某楼层的火情状况、一个地面目标的人员状态），可暂停连续视频流，改为单帧高分辨率快照模式：相机切换至4000×3000（12MP）分辨率拍摄一帧高质量JPEG图像（约500KB-1.5MB），以最高可靠性和低优先级在COFDM链路上传输。快照模式的触发由操作员在态势界面上点击目标位置触发，系统根据点击位置自动计算对应的无人机相机指向和变焦参数（或指派最近且携带所需相机载荷的无人机执行快照任务），快照拍摄到回传显示的端到端延迟约为2-5s（主要是JPEG编码和传输时间）。快照传输期间，视频流暂停或降低至720p低码率模式（1Mbps），为快照释放带宽。
[0528] <Heading 4> 5.4.4  通信车道与QoS管理
[0529] <Normal> QoS策略的设计逻辑，借助车道管理模型来理解。将COFDM链路的物理层带宽视为一条多车道高速公路，不同业务类型运行于不同车道，各自的QoS需求对应着不同的管控参数：带宽上限相当于速度限制，优先级对应超车权限，可靠性要求则对应车道本身的质量等级。通信资源的分配逻辑，由此得以直观呈现。
[0530] <Normal> 车道模型定义了五条逻辑车道。控制车道-实际运行在1.4GHz TDMA链路上，不在COFDM链路中，但在逻辑上属于最高优先级车道，有专用的物理高速公路（独立频段和射频），完全不受COFDM侧拥堵的影响。感知车道-遥测和感知数据的上行通道，分配COFDM总带宽的15%作为保底带宽（在6Mbps低端工作点约为900kbps），链路空闲时可借用其他车道未使用的带宽，但在链路拥堵时必须至少保证保底带宽。遥测车道-遥测数据的上行通道（与感知数据共享COFDM物理链路但在发送队列中优先级高于感知数据），分配保底带宽5%（约300kbps，容纳30机×70B×10Hz=168kbps绰绰有余）。图传车道-视频和快照的上行通道，分配COFDM总带宽的40%作为目标带宽（在QPSK-3/4工作时约7.2Mbps，支持2-3路2Mbps视频流并发），但图传车道为尽力而为（best-effort）-当感知车道和遥测车道需要更多带宽时（如感知数据因复杂场景而暴涨），图传车道的带宽被自动缩减（视频码率从2Mbps降至1Mbps或720p@1Mbps）。日志车道-日志数据的上行和心跳维护通道，无保底带宽，完全使用链路空闲带宽传输，在链路无空闲时暂停传输。车道之间的优先级和带宽保证由COFDM MAC层的发送调度器执行-每100ms根据当前各车道的队列深度和保底/目标带宽需求做一次带宽重分配。

[0533] <Heading 4> 5.4.5  控制链路与数据/图传链路隔离
[0534] <Normal> 控制链路与数据/图传链路的隔离是通信子系统设计中一个不可妥协的架构原则-它不是锦上添花的优化项而是安全底线。隔离在四个层面实现。频段隔离-控制链路1.4GHz，数据/图传链路2.4GHz，利用频率选择性的双工器在射频前端将两个链路分开，单条天线即可同时接收两个频段，互相之间的带外抑制≥60dB（双工器的典型隔离度），控制链路的接收机看不到2.4GHz信号的能量。射频隔离-控制链路和数据链路使用独立的射频前端芯片（独立的LNA、混频器、ADC），即使某一条链路的射频前端因强干扰而饱和，另一条链路不受影响。这一设计的成本是增加了一套射频前端（约15-20g重量和200mW功耗），但换来了硬件级别的故障隔离。协议隔离-两条链路使用完全不同的PHY/MAC协议栈（TDMA/GFSK vs COFDM/AMC），不存在协议栈共享带来的软件耦合和优先级反转风险。管理层隔离-两条链路的配置、监控、故障处理在管理面上分开-控制链路的链路质量告警不影响数据链路的管理状态，反之亦然。操作员在态势界面上可以分开查看两条链路的实时状态（各自的SNR、PER、吞吐量和当前AMC档位），但不允许手动断开控制链路。
[0535] <Heading 3> 5.5 通信可靠性与容错设计
[0536] <Normal> 前面的设计，在链路正常或轻度劣化的条件下是足够的。但应急救援现场的通信环境远非如此理想，局部遮挡、同频干扰、多径衰落、设备故障，任何一项都可能造成链路中断或质量骤降，且往往难以预判。这是通信可靠性与容错设计必须面对的真实约束。核心目标因此被明确为：在单点故障、局部链路中断或全网质量下降的条件下，系统的通信能力逐级退化，而不是整体崩溃。控制指令的传输永远处于最高优先级，安全功能的维持不能以完美通信为前提。
[0537] <Heading 4> 5.5.1  链路质量监测
[0538] <Normal> 链路质量监测依托实时采集的物理层与MAC层指标，并在此基础上划定四个等级的健康状态。物理层侧采集：接收信号强度（RSSI，dBm）、信噪比（SNR，dB）、误比特率（BER，即GFSK解调后硬判决的比特错误比例）。MAC层侧采集：包错误率（PER，CRC校验失败比例）、ARQ重传次数、当前AMC档位（仅适用于COFDM链路）。上述指标由PHY与MAC层硬件寄存器以100ms为周期周期性上报，汇聚至链路质量管理模块后进行综合分析。这一采样频率的设定本身就是一个权衡，过低则响应滞后，过高则增加系统负载，100ms是在两者之间的折中选择。
[0539] <Normal> 链路质量告警分为四个等级，各等级的判定条件与响应动作如下。绿色（正常）：SNR高于AMC档位灵敏度阈值至少6dB裕度，PER低于1%，无需任何降级动作。黄色（注意）：SNR低于阈值但裕度仍为正（0–6dB），PER在1–5%之间，此时触发数据链路的轻度降级，视频码率降低一档，日志暂停传输，控制链路不受影响。橙色（警告）：SNR接近或略低于阈值（裕度-3到0dB），PER在5–20%之间，数据链路进入中度降级状态（感知发送频率降至2Hz，遥测降至2Hz，视频暂停、仅传关键帧），控制链路同步启用ARQ重传机制，最多3次重试。红色（危险）：SNR远低于阈值（裕度低于-3dB），PER超过20%，且控制链路连续3次重试仍未收到ACK，此时触发边缘自治切换（详见第6章），数据链路退化为仅维持心跳与最低限度遥测，1Hz频率，仅发送位置、电量和故障码三个字段。
[0540] <Heading 4> 5.5.2  丢包重传与确认机制
[0541] <Normal> 不同业务的丢包重传策略之所以截然不同，根本原因在于它们的时延预算与可靠性要求本就不在同一量级。控制指令采用停等式ARQ：每发送一条指令，发送方等待ACK，超时未收到则重传，最多3次，总重传预算约90ms，仍低于P1指令100ms的边界。3次重传后仍无ACK，判定该指令发送失败，进而触发通信异常安全策略（通常为激活边缘自治模式或将该无人机标记为通信中断）。遥测与感知数据则不采用逐包ACK，状态数据是周期性的，单包丢失可由下一包覆盖，而ACK本身的额外开销（约20字节/包，在30机×10Hz场景下达48kbps）远大于其实际价值，引入它得不偿失。视频流走的是另一条路：在H.265码流中插入约5%的FEC冗余修复包，接收端在丢包率低于5%的情况下可直接恢复丢失帧，无需触发重传。日志数据对时延不敏感但要求完整，通过TCP连接在后台慢速上传，TCP自身的拥塞控制与重传机制已完全满足需求，无需额外设计。
[0542] <Heading 4> 5.5.3  节点掉线与簇头切换
[0543] <Normal> 节点掉线检测通过心跳超时机制实现。每架无人机以1Hz频率发送心跳消息，地面基站和簇头各自维护一个心跳计时器-每收到该无人机的心跳消息时计时器归零，每经过1s计时器递增。连续3次心跳超时（即3s内未收到任何心跳消息）触发节点掉线判定。心跳消息的丢失可能由三个原因导致：无人机故障坠毁（心跳永久停止）、无人机通信设备故障（飞控正常但数传模块失效）、无人机飞入通信盲区（飞控和通信均正常但信号被遮挡）。仅凭心跳超时无法区分这三个原因-系统默认假定为通信盲区（最可恢复的情形），执行5.5.4节的三断安全策略，同时持续监听该机的心跳恢复。
[0544] <Normal> 簇头切换（5.2.1节已定义备簇头接管机制）在工程上需要解决一个数据连续性挑战：簇头切换期间，原簇头尚未转发的成员状态数据和正在排队等待发送的指令如何处理。解决方案为：备簇头在每次收到正簇头的调度表镜像时同步缓存簇内所有成员的最近一次状态数据，接管时这些缓存的状态数据作为切换后的初始状态基准，避免切换导致地面平台对该簇成员的状态跟踪中断。正在排队等待发送的指令-如果是指令是从地面基站下发到正簇头且尚未转发至目标成员的，地面基站侧维护一个待确认指令队列，在收到簇头切换通知后自动对未确认的指令重新下发至备簇头；如果是成员上行数据，成员在发送时隙中未收到正簇头ACK时（因簇头已离线），自动将数据缓存并在备簇头发送的新调度表的第一个上行时隙中重新发送。
[0545] <Heading 4> 5.5.4  通信异常时的安全策略
[0546] <Normal> 通信异常的安全策略按照从轻微到严重分为四级响应。策略一（链路质量下降，黄色告警）-仅调整数据链路的QoS降级策略（降低视频和感知的频率/码率），控制链路维持正常，无人机继续执行原任务，操作员收到黄色提示但不强制干预。策略二（控制链路轻度劣化，橙色告警）-控制链路启用ARQ重传，调度器暂停非紧急的新任务分配（避免在通信不稳定时引入新航线增加不确定性），已在执行任务的无人机继续执行但操作员被告知通信风险。策略三（控制链路严重劣化，红色告警，某无人机控制链路中断）-受影响无人机自动切换至边缘自治模式（第6章），基于本地栅格子集和机载感知执行保守的局部消解，无人机继续执行当前任务（如果链路在规定时间-默认30s-内恢复，自动切换回集中式调度）或自动返航（如果链路持续中断超过30s且机电量低于50%-保证有充足电量在无通信引导条件下安全返航至回收点）。策略四（地面基站与全部无人机通信中断，全网失联）-全部无人机自动执行预先设置的失联安全策略：保持当前高度和位置悬停等待30s→如果30s内恢复则返回集中调度→如果30s未恢复则各机沿预置返航航线（在任务开始前下载到每架无人机的飞控中，不依赖于通信链路）自动返航至回收点→在回收点上空悬停等待指令。策略四在救援现场是极端情况（例如地面基站被灾害二次冲击摧毁），但它必须在系统设计阶段被考虑到、设计好并在外场演练中验证。
[0547] <Normal> 需要特别明确一个架构边界：分布式软总线承担设备发现、时序协同和上层数据协同的职能-它保证无人机、网关和地面站之间知道彼此存在、知道彼此的状态、能以统一的时序协作。但软总线不直接承载10km的无线通信链路-无线信号的传播、调制、解调、同步、纠错等物理层工作由TDMA/GFSK和COFDM/OFDM的独立物理链路完成。软总线运行在这些物理链路之上，将它抽象为统一的数据通道供上层模块使用。这个边界必须清晰划定-否则会误导验收评审者以为分布式软总线本身提供了10km的无线通信能力。
[0548] <Heading 4> 5.5.5  通信子系统实施步骤
[0549] <Normal> 通信子系统的实施划分为四个阶段，各阶段目标层层递进。第一阶段（第1至4月），核心任务是把物理层的关键参数从设计图纸落到可测试的原型上。具体来说，需在实验室环境中以软件无线电平台（USRP或AD936x评估板均可）搭建TDMA帧结构原型与COFDM收发链路原型，验证30ms帧/36时隙的PHY层闭环行为，以及64子载波COFDM的AMC自适应切换功能。GFSK调制解调的FPGA/ASIC实现方案选型和初步逻辑设计也在本阶段完成。此外，簇头选举算法的仿真验证同步推进，在MATLAB或Python环境中模拟30架无人机的位置变化与电量消耗，重点考察四维加权选举的稳定性和备份接管延迟。这一阶段说到底是在为后续集成打底，任何在仿真层面未解决的参数问题，到了联调阶段代价只会更高。
[0550] <Normal> 第二阶段（第5至8月），重心从物理层转向协议栈的完整实现。需要完成的模块包括：TDMA MAC层（时隙分配与调度器）、COFDM MAC层（AMC控制环路与QoS车道调度器）、简化AODV路由协议，以及簇头选举与备份协议。要在不少于5个物理节点的自组网测试床上跑通基本功能，覆盖簇建立、簇头切换、多跳路由和双链路隔离这几个关键场景。H.265视频编码器在机载嵌入式平台（海思或RK3588编码器IP）上的移植与实时编码验（2Mbps@1080p@30fps）也在本阶段收口。
[0551] <Normal> 第三阶段（第9至14月），进入外场联调，规模逐步扩展。5机阶段的重点是TDMA时隙分配与多机时钟同步精度，使用GNSS授时基准测试仪测量各节点时钟偏差，这是后续规模扩展的基础。10机阶段转向COFDM链路在多机并发上行条件下的吞吐量表现，需在10km轴线内各距离点实测链路SNR与实际吞吐量，绘制距离-吞吐量曲线，AMC档位分布同步记录。20机阶段专项验证双链路隔离的有效性：在视频流满载COFDM链路（4路2Mbps视频并发）的条件下，观测控制链路的延迟与丢包率是否受到影响，这是判断隔离机制是否真正有效的直接判据。30机阶段做全系统通信压力测试，30机同时上行遥测、感知与4路视频，测量COFDM链路总吞吐量与各车道保底带宽的兑现情况。
[0552] <Normal> 第四阶段（第15至18月），综合演练与验收。核心场景是模拟三断条件下的全系统通信运行：关闭公网与市电，仅靠自组网和系统自带电源维持30架无人机在10km轴线内的全部通信，连续运行不低于2小时，全时段记录链路质量日志，包括SNR、PER、AMC档位分布、各车道吞吐量及各优先级指令的端到端延迟统计，最终生成通信子系统性能评估报告。除此之外，还需验证通信异常四级安全策略的逐级触发与恢复，演练中有意注入链路劣化（使用可调衰减器逐步降低信号强度），考察系统能否按策略正确逐级降级并恢复。这是整个子系统可靠性的最终背书。

    软总线的会话管理覆盖从设备互不可见到全双工数据通信的完整过程，可拆分为四个阶段来理解。第一阶段是设备发现（Discovery）。每个节点的软总线服务启动后，会在TDMA的预留广播时隙内向网络发送一条Discovery报文。这是广播时隙，不参与业务时隙的竞争。报文携带三类信息：节点类型（机载/网关/地面站/基站代理）、全局唯一标识（设备序列号或MAC地址的SHA-256哈希前16字节），以及初始能力摘要（支持的协议版本、最大消息长度、安全能力等）。收到报文的节点将发送方加入本地设备列表并回复Ack。发现延迟由广播时隙的出现周期决定，时隙每100ms出现一次，首次广播与时隙之间的平均等待时间约为50ms，这也就是发现延迟的期望值。会话建立（Session Establishment）紧随其后。发现完成后，发起方通常是地面站或网关，向目标节点发送SessionRequest报文，双方经由三次握手（Request→Accept→Confirm）协商会话参数：MTU由底层COFDM链路的当前AMC档位决定，64QAM时为1500字节，QPSK时降至256字节；心跳间隔默认1秒，在TDMA控制时隙中发送，不占用业务时隙；会话超时默认5秒，5秒内未收到心跳即判定会话失效并触发自动重连。三次握手中任何一步超时（超时窗口=2×心跳间隔+往返时间估计），会话建立即告失败并回退至Discovery状态。它是在主动规避半开连接（half-open connection）长期挂在会话表中的问题。第三阶段是状态同步（State Sync）。会话建立后，双方立即交换初始状态：机载端全量上报当前的DroneState和DroneCapability（约2-5KB），地面端则下发全局配置，空域边界多边形顶点列表、禁飞区geofence定义、当前任务列表，全量约3-10KB。初始同步完成后，后续更新切换为增量推送，仅发送变化字段（如位置从(x1,y1,z1)更新为(x2,y2,z2)时只推送位置字段的新值），单条增量报文约50-200字节。这里有一个数字可以说明问题：若不使用增量，30架飞机以10Hz频率并发上报、每次2KB，带宽消耗约为600KB/s，折合近4.8Mbps；增量方案在30架机群并发场景下的节约效果，由此可见。引入增量机制后，典型传输速率降至约30架×10Hz×100B=30KB/s，折合约240Kbps，这一数字仅占COFDM控制链路最低AMC档（2Mbps BPSK）的12%左右，带宽余量相当可观。阶段四是会话维护（Session Maintenance）。会话激活之后，双方按心跳间隔周期性地交换HeartbeatMsg。心跳报文的设计并非单纯的存活探测，它同时承载轻量级的增量状态更新，将位置、电量等高频变化字段的当前值直接嵌入报文，从而避免为状态同步额外发包。这个设计决策背后的逻辑是减少报文数量。当连续N个心跳周期（N=3，默认值）未收到对端响应时，软总线判定会话超时，随即触发拆除流程：发出SessionClose通知（尽力而为，UDP层面不保证送达），释放会话ID，向所有订阅了该设备Topic的上层模块推送设备离线事件，并将注册表中的设备状态更新为OFFLINE。会话超时并不触发注册信息的删除。设备能力、协议类型、历史连接记录等仍保留在注册表中，这一保留策略的意义，在设备重新上线时才完全显现：重连方可跳过发现与注册阶段，直接进入会话恢复。快速重连的延迟大致等于SessionRequest-Accept-Confirm三次握手所需时间，约3×RTT，实测在15至30ms之间。

任务协议翻译层做的事，是把地面平台的统一任务接口：takeoff、land、goto、upload_mission等翻译为对应的MAVLink消息序列，并处理ACK/NACK反馈。这个翻译层的存在，本质上是在抹平上层接口与底层协议之间的语义落差。版本兼容的处理方式：同时支持MAVLink v1和v2，以消息ID为区分依据，ID不超过255的消息在两个版本中格式一致，v2的扩展帧仅在ID大于255时启用；心跳消息中携带的版本信息作为辅助判定依据。这个设计的好处是向下兼容的代价很小。

原子性是第一个原则。每个接口只完成一个明确的飞行控制语义动作，不存在执行复杂序列这类复合接口，复杂任务的编排，是上层调度器的职责，而非接口本身应当承担的负担。幂等性紧随其后：同一指令在短时间内重复下发不产生副作用，以连续两次takeoff为例，第二次调用直接返回已处于飞行状态，不会重新触发起飞序列。第三个原则是超时可中断，每个接口调用携带超时参数，超时后网关主动放弃等待并向上层返回超时错误。

struct DroneState {
    uint16_t drone_id;            // 设备唯一ID (注册时分配)
    uint8_t  drone_type;          // 类型: 0=侦察 1=投送 2=中继 3=其他
    // --- 位置与运动 (ENU坐标系, 单位: m, m/s, rad) ---
    double   pos_x, pos_y, pos_z; // ENU位置 (WGS84投影至ENU)
    double   vel_x, vel_y, vel_z; // ENU速度
    double   roll, pitch, yaw;    // 姿态角 (欧拉角 RPY)
    double   ang_vel_x, ang_vel_y, ang_vel_z; // 角速度
    // --- 导航状态 ---
    uint8_t  nav_status;          // 0=GNSS固定解 1=GNSS浮点解
                                  // 2=视觉里程计 3=ESKF融合 4=惯性推测
    float    nav_accuracy;        // 定位精度估计 (m, CEP)
    float    heading_accuracy;    // 航向精度估计 (deg, 1σ)
    // --- 能源 ---
    uint8_t  battery_soc;         // 剩余电量百分比 (0-100%)
    float    battery_voltage;     // 电池电压 (V)
    float    current_draw;        // 当前电流 (A)
    int16_t  est_remaining_s;     // 估计剩余飞行时间 (s)
    // --- 飞行模式与任务 ---
    uint8_t  flight_mode;         // 0=待命 1=自动任务 2=手动操控
                                  // 3=RTL返航 4=降落 5=紧急
    uint16_t current_mission_id;  // 当前执行的任务ID (0=无任务)
    uint8_t  mission_status;      // 0=待执行 1=执行中 2=已完成 3=异常
    // --- 健康与状态 ---
    uint8_t  health_flags;        // bit0=飞控 bit1=感知 bit2=通信
                                  // bit3=电源 bit4=执行机构 (1=正常 0=异常)
    uint16_t fault_code;          // 详细故障码 (0=无故障)
    // --- 链路 ---
    int8_t   link_rssi;           // 接收信号强度 (dBm)
    uint8_t  link_snr;            // 信噪比 (dB)
    float    link_delay_ms;       // 端到端延迟 (ms)
    // --- 时间戳 ---
    uint64_t timestamp_ms;        // 消息生成时间 (系统时间, ms)
};
struct DroneCommand {
    uint16_t drone_id;            // 目标无人机ID
    uint8_t  cmd_type;            // 0=航点 1=速度 2=上传航线
                                  // 3=起飞 4=降落 5=RTL 6=紧急悬停
    uint8_t  priority;            // 0=紧急 1=高 2=常规 3=低
    // --- 航点指令参数 (cmd_type=0) ---
    double   wp_x, wp_y, wp_z;    // 目标航点 (ENU)
    float    wp_speed;            // 巡航速度 (m/s)
    float    wp_accept_radius;    // 到达判定半径 (m)
    // --- 速度指令参数 (cmd_type=1) ---
    double   vel_cmd_x, vel_cmd_y, vel_cmd_z; // 期望速度 (m/s)
    float    vel_cmd_yaw;         // 期望航向角 (rad)
    uint16_t vel_cmd_duration_ms; // 持续时长 (ms, 0=持续至新指令)
    // --- 航线上传参数 (cmd_type=2) ---
    uint8_t  wpt_count;           // 路径点数量 (≤50)
    // 路径点数组通过可变长度字段附加在消息尾部
    // --- 通用 ---
    uint32_t cmd_id;              // 指令ID (递增, 用于ACK对账)
    uint8_t  require_ack;         // 0=无需ACK 1=需ACK确认
    uint64_t expire_ms;           // 指令过期时间 (超时未执行则丢弃)
};
struct DroneCapability {
    uint16_t drone_id;            // 所属无人机ID
    // --- 飞行性能 ---
    float    max_speed;           // 最大平飞速度 (m/s)
    float    max_climb_rate;      // 最大爬升率 (m/s)
    float    max_altitude;        // 最大飞行高度 (m, 相对起飞点)
    float    max_endurance_s;     // 最大续航时间 (s, 标准工况)

constexpr uint16_t TDMA_FRAME_MS    = 30;    // 帧周期
constexpr uint16_t TDMA_SLOTS       = 36;    // 时隙数
constexpr uint16_t TDMA_GUARD_US    = 50;    // 保护间隔 (us)
constexpr uint16_t TDMA_SLOT_US     = (TDMA_FRAME_MS * 1000) / TDMA_SLOTS;  // 每时隙约833us

enum class SlotDirection : uint8_t {
    UPLINK   = 0,  // 无人机 → 基站
    DOWNLINK = 1,  // 基站 → 无人机
    GUARD    = 2   // 保护时隙
};

struct SlotAllocation {
    uint16_t      node_id;        // 占用节点ID (0=基站, 1-30=无人机)
    SlotDirection direction;
    uint8_t       priority;       // 0=紧急 1=高 2=常规 (继承自数据QoS)
};

struct TdmaFrame {
    uint32_t                          frame_id;    // 帧序号 (递增)
    uint64_t                          start_us;    // 帧起始时刻 (系统时间us)
    std::array<SlotAllocation, TDMA_SLOTS> slots;  // 时隙分配表
};

#include <cmath>

struct NodeDemand {
    uint16_t node_id;
    SlotDirection dir;
    float    rate_kbps;   // 需求速率
    uint8_t  priority;
    uint16_t allocated = 0;  // 已分配时隙数
};

// 单时隙容量: 833us × 200kbps(GFSK) ≈ 167 bits ≈ 21 bytes/时隙
constexpr float SLOT_CAPACITY_KBPS = (21.0f * 8.0f) / (TDMA_FRAME_MS / 1000.0f);  // ≈5.6kbps

TdmaFrame greedySlotAllocation(const std::vector<NodeDemand>& demands) {
    TdmaFrame frame{};
    frame.frame_id = 0;  // 由上层维护

    // 复制并按优先级+速率需求降序
    auto queue = demands;
    std::sort(queue.begin(), queue.end(),
        [](const NodeDemand& a, const NodeDemand& b) {
            if (a.priority != b.priority) return a.priority < b.priority;
            return a.rate_kbps > b.rate_kbps;
        });

    uint16_t slot_idx = 0;
    // 前两个时隙固定为保护时隙 (帧边界)
    frame.slots[0].direction = SlotDirection::GUARD;
    frame.slots[TDMA_SLOTS-1].direction = SlotDirection::GUARD;

    // 贪心分配: 为每个节点分配足够满足其速率需求的时隙数
    for (auto& dm : queue) {
        uint16_t needed = static_cast<uint16_t>(
            std::ceil(dm.rate_kbps / SLOT_CAPACITY_KBPS));
        needed = std::max(needed, uint16_t(1));  // 至少1个时隙

        for (uint16_t s = 1; s < TDMA_SLOTS - 1 && dm.allocated < needed; ++s) {
            if (frame.slots[s].node_id == 0) {  // 空闲时隙
                frame.slots[s].node_id    = dm.node_id;
                frame.slots[s].direction  = dm.dir;
                frame.slots[s].priority   = dm.priority;
                dm.allocated++;
            }
        }
    }
    return frame;
}

// 估算节点N的平均时隙等待延迟 (ms)
float estimateWaitDelay(const TdmaFrame& frame, uint16_t node_id) {
    // 统计该节点的时隙间隙，取平均间隔 / 2 作为期望等待时间
    // (假设数据包到达时间在帧内均匀分布)
    float avg_interval_ms = static_cast<float>(TDMA_FRAME_MS) / TDMA_SLOTS;
    return avg_interval_ms * 0.5f;  // ~0.42ms for 30ms/36slots
}

enum class AmcLevel : uint8_t {
    BPSK  = 0,   // SNR≥3dB,  吞吐量2Mbps,  BER<1e-6
    QPSK  = 1,   // SNR≥7dB,  吞吐量4Mbps
    QAM16 = 2,   // SNR≥12dB, 吞吐量8Mbps
    QAM64 = 3    // SNR≥18dB, 吞吐量12Mbps (需要最高SNR)
};

struct AmcState {
    AmcLevel level;
    float    snr_db;          // 当前估计SNR
    float    throughput_mbps;  // 物理层吞吐量
    float    target_ber;      // 目标误码率
};

// cofdm_amc.cpp — AMC切换逻辑
#include cofdm_amc.h
#include <algorithm>

constexpr float AMC_THRESHOLDS[]  = {3.0f, 7.0f, 12.0f, 18.0f};  // SNR门限 (dB)
constexpr float AMC_HYSTERESIS_DB = 1.5f;   // 迟滞 (防止来回切换)
constexpr float AMC_THROUGHPUT[]  = {2.0f, 4.0f, 8.0f, 12.0f}; // Mbps

class CofdmAmcController {
public:
    explicit CofdmAmcController(float hysteresis_db = AMC_HYSTERESIS_DB)
        : hysteresis_db_(hysteresis_db) {}

    // 基于当前SNR估计选择最优AMC档位 (每帧边界调用一次)
    AmcState selectLevel(float estimated_snr_db) {
        AmcLevel new_level = current_.level;
        float best_snr = estimated_snr_db;

        // 尝试升档 (当前档位SNR超过上一档门限+迟滞)
        if (static_cast<int>(current_.level) < 3) {
            float next_thresh = AMC_THRESHOLDS[static_cast<int>(current_.level) + 1];
            if (best_snr > next_thresh + hysteresis_db_) {
                new_level = static_cast<AmcLevel>(static_cast<int>(current_.level) + 1);
            }
        }
        // 尝试降档 (当前档位SNR低于当前档门限-迟滞)
        if (static_cast<int>(current_.level) > 0) {
            float curr_thresh = AMC_THRESHOLDS[static_cast<int>(current_.level)];
            if (best_snr < curr_thresh - hysteresis_db_) {
                new_level = static_cast<AmcLevel>(static_cast<int>(current_.level) - 1);
            }
        }

        current_.level            = new_level;
        current_.snr_db           = best_snr;
        current_.throughput_mbps  = AMC_THROUGHPUT[static_cast<int>(new_level)];
        current_.target_ber       = (new_level == AmcLevel::BPSK) ? 1e-6f : 1e-5f;
        return current_;
    }

    // 理论BER估计 (AWGN信道, QPSK/QAM Gray映射, 简化公式)
    static float estimateBerAwgn(float snr_db, AmcLevel level) {
        float snr_linear = std::pow(10.0f, snr_db / 10.0f);
        int M = 2 << static_cast<int>(level);  // 2,4,16,64
        // 近似: Q(sqrt(3*SNR/(M-1))) * 4*(1-1/sqrt(M))/log2(M)
        float arg = std::sqrt(3.0f * snr_linear / (M - 1));
        float q_val = 0.5f * std::erfc(arg / std::sqrt(2.0f));
        return q_val * (4.0f * (1.0f - 1.0f / std::sqrt(M))) / std::log2(M);
    }

private:
    float    hysteresis_db_;
    AmcState current_{AmcLevel::BPSK, 20.0f, 2.0f, 1e-6f};
};

using VoxelKey = std::tuple<int32_t, int32_t, int32_t, int64_t>;

struct VoxelKeyHash {
    size_t operator()(const VoxelKey& k) const {
        auto [x, y, z, t] = k;
        return ((size_t)x * 73856093) ^ ((size_t)y * 19349663)
             ^ ((size_t)z * 83492791) ^ ((size_t)t * 39916801);
    }
};

struct ConflictResult {
    uint16_t drone_a;
    uint16_t drone_b;
    int64_t  conflict_time_ms;    // 冲突时间
    float    closest_dist_m;       // 最接近距离
    int      level;               // 0=无 1=潜在(预测) 2=确认
};

// 检测所有在线无人机对之间的冲突 (OpenMP并行遍历)
std::vector<ConflictResult> detectAllConflicts(
    const Grid4D& grid,
    const std::vector<uint16_t>& online_drone_ids,
    float safety_xy, float safety_z)
{
    const size_t N = online_drone_ids.size();
    const size_t num_pairs = N * (N - 1) / 2;

    // 预构建机对列表 (便于OpenMP并行)
    std::vector<std::pair<uint16_t, uint16_t>> pairs;
    pairs.reserve(num_pairs);
    for (size_t i = 0; i < N; ++i) {
        for (size_t j = i + 1; j < N; ++j) {
            pairs.emplace_back(online_drone_ids[i], online_drone_ids[j]);
        }
    }

    std::vector<ConflictResult> results(pairs.size());

    // OpenMP并行: 每对独立检测, 无数据依赖
    #pragma omp parallel for schedule(dynamic, 8)
    for (size_t k = 0; k < pairs.size(); ++k) {
        auto [ida, idb] = pairs[k];
        results[k].drone_a = ida;
        results[k].drone_b = idb;
        results[k].level   = 0;

        // 快速初筛: 两机是否有体素占用交集?
        if (!grid.hasVoxelOverlap(ida, idb)) {
            continue;  // 无交集 → 无冲突
        }

        // 详细检测: 获取两机的体素占用列表并找时间重叠
        auto vox_a = grid.getOccupiedVoxels(ida);
        auto vox_b = grid.getOccupiedVoxels(idb);

        // 找到时间维度重叠的体素对, 计算最接近空间距离
        float min_dist = std::numeric_limits<float>::max();
        int64_t conflict_t = 0;

        for (const auto& [ax, ay, az, at] : vox_a) {
            for (const auto& [bx, by, bz, bt] : vox_b) {
                if (at != bt) continue;  // 不同时间步, 不可能冲突

                // 体素中心距离 (近似, 实际应通过轨迹插值)
                float dx = (ax - bx) * GRID_RES_M;
                float dy = (ay - by) * GRID_RES_M;
                float dz = (az - bz) * GRID_RES_M;
                float d  = std::sqrt(dx*dx + dy*dy + dz*dz);

                if (d < min_dist) {
                    min_dist  = d;
                    conflict_t = at * GRID_TIME_STEP;
                }
            }
        }

        // 判定冲突等级
        if (min_dist < std::numeric_limits<float>::max()) {
            float sep = std::min(safety_xy, safety_z);
            results[k].closest_dist_m    = min_dist;
            results[k].conflict_time_ms  = conflict_t;
            results[k].level = (min_dist < sep) ? 2 : 1;  // 确认/潜在
        }
    }

    // 过滤: 仅保留有冲突的结果
    std::vector<ConflictResult> filtered;
    filtered.reserve(results.size());
    for (const auto& r : results) {
        if (r.level > 0) filtered.push_back(r);
    }
    return filtered;
}

enum class ResolvePrimitive {
    STOP = 0,       // 悬停等待 (零速度指令)
    REROUTE = 1,    // 路径重规划
    ALTITUDE = 2    // 高度层调整 (上升/下降至另一高度层)
};

struct ResolveCandidate {
    uint16_t          drone_id;
    ResolvePrimitive  primitive;
    std::vector<std::array<float,3>> new_waypoints;  // 新路径点序列
    float             cost_path;      // 路径代价 (额外飞行距离)
    float             cost_delay;     // 时间延迟代价
    float             cost_energy;    // 能耗代价
    float             safety_gain;    // 安全收益 (负成本, 越大越好)
    float             total_cost;     // 加权总成本
};

// 生成三个消解候选方案 (每个原语一个方案)
std::vector<ResolveCandidate> generateCandidates(
    const ConflictResult& conflict,
    const Grid4D& grid,
    const std::vector<std::array<float,3>>& traj_a,  // 无人机A的参考轨迹
    const std::vector<std::array<float,3>>& traj_b)  // 无人机B的参考轨迹
{
    std::vector<ResolveCandidate> candidates;

    // 候选1: STOP — 无人机A在冲突点前方悬停
    {
        ResolveCandidate c;
        c.drone_id   = conflict.drone_a;
        c.primitive  = ResolvePrimitive::STOP;
        // 路径: 正常飞行至冲突前2s → 悬停5s → 继续
        c.cost_path    = 0.0f;    // 无额外距离
        c.cost_delay   = 5.0f;     // 5s等待
        c.cost_energy  = 5.0f * 200.0f / 3600.0f; // 悬停功耗率
        c.safety_gain  = 1.0f;     // 最高安全收益 (完全避让)
        candidates.push_back(c);
    }

    // 候选2: REROUTE — 搜索新路径绕开冲突体素
    {
        ResolveCandidate c;
        c.drone_id   = conflict.drone_a;
        c.primitive  = ResolvePrimitive::REROUTE;
        // 在自由体素图上搜索 (start=当前位置, goal=原航线终点)
        // (实现省略 — 标准栅格, 代价=g+h, h=欧氏距离到目标)
        auto new_path = astarSearch(
            traj_a.front(),        // start pos
            traj_a.back(),         // goal pos
            grid,                  // 4D占用栅格 (障碍体素=被其他机占用)
            conflict.drone_a
        );
        c.new_waypoints = new_path;
        c.cost_path    = pathLength(new_path) - pathLength(traj_a);
        c.cost_delay   = c.cost_path / 10.0f;  // 按10m/s巡航速度估算
        c.cost_energy  = c.cost_path * 1.5f;    // 能耗因子
        c.safety_gain  = 0.8f;
        candidates.push_back(c);
    }

    // 候选3: ALTITUDE — 将A机升高5m (高度层分离)
    {
        ResolveCandidate c;
        c.drone_id   = conflict.drone_a;
        c.primitive  = ResolvePrimitive::ALTITUDE;
        // 将整个航线的Z轴偏移+5m
        auto shifted_traj = traj_a;
        for (auto& wp : shifted_traj) wp[2] += 5.0f;
        c.new_waypoints = shifted_traj;
        c.cost_path    = 0.0f;
        c.cost_delay   = 0.0f;
        c.cost_energy  = 5.0f * 20.0f / 3600.0f;  // 爬升5m额外能耗
        c.safety_gain  = 0.7f;
        candidates.push_back(c);
    }

    return candidates;
}

// 成本评估: 加权总分 (权重可配置, 以下是默认值)
float evaluateCost(const ResolveCandidate& c,
                   float w_path=1.0f, float w_delay=1.5f,
                   float w_energy=0.5f, float w_safety=-2.0f) {
    return w_path * c.cost_path
         + w_delay * c.cost_delay
         + w_energy * c.cost_energy
         + w_safety * c.safety_gain;  // 安全收益为负成本
}

// 安全过滤层: 4条硬约束规则 (纯C逻辑, <2KB代码)
struct SafetyFilter {
    // 规则1: 方案是否导致无人机进入禁飞区?
    static bool checkNoFlyZone(const ResolveCandidate& c,
                               const std::vector<std::array<float,2>>& no_fly_zones) {
        for (const auto& wp : c.new_waypoints) {
            if (pointInPolygon(wp[0], wp[1], no_fly_zones)) return false;
        }
        return true;
    }

class MavlinkAdapter : public IProtocolAdapter {
public:
    std::string protocolName() const override { return MAVLink_v2; }

    // 标准指令 → MAVLink消息 (示例: 航点指令 → MISSION_ITEM_INT)
    std::vector<uint8_t> encodeCommand(const DroneCommand& cmd) override {
        mavlink_message_t msg;
        uint8_t buf[MAVLINK_MAX_PACKET_LEN];

        if (cmd.cmd_type == 0) {  // 航点指令
            mavlink_msg_mission_item_int_pack(
                1, 0, &msg,         // sysid=1, compid=0 (地面站)
                cmd.drone_id, 0,    // target_system, target_component
                0,                  // seq (由无人机维护)
                MAV_FRAME_GLOBAL_RELATIVE_ALT,
                MAV_CMD_NAV_WAYPOINT,  // 航点命令
                0, 0,               // current, autocontinue
                0.0f, 0.0f, 0.0f,   // param1-3 (未使用)
                static_cast<float>(cmd.wp[0] * 1e7),  // x (1e7度 → int32)
                static_cast<float>(cmd.wp[1] * 1e7),  // y
                cmd.wp[2]            // z (相对高度, m)
            );
        } else if (cmd.cmd_type == 5) {  // RTL返航
            mavlink_msg_command_long_pack(
                1, 0, &msg, cmd.drone_id, 0,
                MAV_CMD_NAV_RETURN_TO_LAUNCH, 0,
                0, 0, 0, 0, 0, 0, 0
            );
        }
        // ... 其他指令类型

        uint16_t len = mavlink_msg_to_send_buffer(buf, &msg);
        return std::vector<uint8_t>(buf, buf + len);
    }

    // MAVLink消息 → 标准DroneState (多消息聚合)
    DroneState decodeState(const std::vector<uint8_t>& raw) override {
        mavlink_message_t msg;
        mavlink_status_t status;

        // 解析MAVLink帧
        for (size_t i = 0; i < raw.size(); ++i) {
            if (mavlink_parse_char(MAVLINK_COMM_0, raw[i], &msg, &status)) {
                return updateStateFromMsg(msg);
            }
        }
        return current_state_;  // 返回最近一次完整状态
    }

    // 从MAVLink心跳+版本消息提取能力声明
    DroneCapability extractCapability(const std::vector<uint8_t>& raw) override {
        DroneCapability cap{};
        // MAVLink HEARTBEAT → type/autopilot/base_mode
        // AUTOPILOT_VERSION → 飞控能力标志位
        // (简化实现: 假设为通用MAVLink无人机, 填充默认能力)
        cap.max_speed       = 15.0f;
        cap.max_altitude    = 500.0f;
        cap.max_endurance_s = 1800.0f;
        cap.supports_vel_ctrl  = 1;
        cap.supports_pos_ctrl  = 1;
        cap.supports_mission   = 1;
        cap.protocol_type      = 0;  // MAVLink
        return cap;
    }

    bool isLikelyThisProtocol(const std::vector<uint8_t>& sample) override {
        // MAVLink v2帧头: 0xFD
        return !sample.empty() && sample[0] == 0xFD;
    }

    std::vector<uint8_t> generateHeartbeat() override {
        mavlink_message_t msg;
        uint8_t buf[MAVLINK_MAX_PACKET_LEN];
        mavlink_msg_heartbeat_pack(
            1, MAV_COMP_ID_MISSIONPLANNER, &msg,
            MAV_TYPE_GCS, MAV_AUTOPILOT_INVALID,
            MAV_MODE_FLAG_CUSTOM_MODE_ENABLED, 0,
            MAV_STATE_ACTIVE
        );
        uint16_t len = mavlink_msg_to_send_buffer(buf, &msg);
        return std::vector<uint8_t>(buf, buf + len);
    }

private:
    DroneState current_state_;

    DroneState updateStateFromMsg(const mavlink_message_t& msg) {
        switch (msg.msgid) {
            case MAVLINK_MSG_ID_GLOBAL_POSITION_INT: {
                mavlink_global_position_int_t pos;
                mavlink_msg_global_position_int_decode(&msg, &pos);
                current_state_.pos[0] = pos.lat / 1e7;   // lat (deg)
                current_state_.pos[1] = pos.lon / 1e7;   // lon (deg)
                current_state_.pos[2] = pos.relative_alt / 1000.0f; // mm→m
                break;
            }
            case MAVLINK_MSG_ID_ATTITUDE: {
                mavlink_attitude_t att;
                mavlink_msg_attitude_decode(&msg, &att);
                current_state_.att[0] = att.roll;
                current_state_.att[1] = att.pitch;
                current_state_.att[2] = att.yaw;
                break;
            }
            case MAVLINK_MSG_ID_SYS_STATUS: {
                mavlink_sys_status_t sys;
                mavlink_msg_sys_status_decode(&msg, &sys);
                current_state_.battery_soc = sys.battery_remaining;
                break;
            }
            // ... 其他消息类型
        }
        return current_state_;
    }

struct FrameFeatures {
    std::map<uint8_t, int> header_counts;  // 帧头字节→出现次数
    int len_min, len_max, len_median;
    int est_hb_ms;        // 估计心跳间隔
    int frame_count;
};

// 从报文序列提取特征 (≥10帧)
FrameFeatures extractFeatures(const std::vector<std::vector<uint8_t>>& frames,
                               const std::vector<int64_t>& timestamps_ms) {
    FrameFeatures feat{};
    feat.frame_count = frames.size();

    std::vector<int> lengths, intervals;
    // 帧头统计 & 帧长收集
    for (const auto& f : frames) {
        if (!f.empty()) feat.header_counts[f[0]]++;
        lengths.push_back(static_cast<int>(f.size()));
    }
    std::sort(lengths.begin(), lengths.end());
    feat.len_min    = lengths.front();
    feat.len_max    = lengths.back();
    feat.len_median = lengths[lengths.size() / 2];

    // 心跳间隔估计: 取最常见的报文间隔
    for (size_t i = 1; i < timestamps_ms.size(); ++i) {
        int dt = static_cast<int>(timestamps_ms[i] - timestamps_ms[i-1]);
        if (dt > 200 && dt < 5000) intervals.push_back(dt);
    }
    if (!intervals.empty()) {
        std::sort(intervals.begin(), intervals.end());
        feat.est_hb_ms = intervals[intervals.size() / 2];
    }
    return feat;
}

// 模板匹配: 计算特征与每个模板的匹配得分
std::pair<std::string, float> matchTemplate(const FrameFeatures& feat) {
    std::string best_name = Unknown;
    float       best_score = 0.0f;

    for (const auto& tmpl : TEMPLATES) {
        float score = 0.0f;

        // 帧头匹配 (+0.4)
        for (auto hb : tmpl.header_bytes) {
            if (feat.header_counts.count(hb)) {
                score += 0.4f;
                break;
            }
        }
        // 帧长范围匹配 (+0.3)
        if (feat.len_median >= tmpl.len_range.first
            && feat.len_median <= tmpl.len_range.second) {
            score += 0.3f;
        }
        // 心跳间隔匹配 (+0.2, 容差<20%)
        if (feat.est_hb_ms > 0) {
            float err = std::abs(feat.est_hb_ms - tmpl.hb_interval_ms)
                        / static_cast<float>(tmpl.hb_interval_ms);
            if (err < 0.2f) score += 0.2f;
            else if (err < 0.5f) score += 0.1f;
        }

        if (score > best_score) {
            best_score = score;
            best_name  = tmpl.name;
        }
    }

    if (best_score < 0.5f) best_name = Unknown;
    return {best_name, std::min(best_score, 0.95f)};

from sklearn.cluster import DBSCAN

# ------- 步骤1: 地面分割 (Patchwork++简化) -------
def segment_ground(pts_xyz, grid_size=0.5, thresh=0.2):
    极坐标栅格地面分割
    r = np.sqrt(pts_xyz[:,0]**2 + pts_xyz[:,1]**2)
    az = np.arctan2(pts_xyz[:,1], pts_xyz[:,0])
    n_r = max(1, int(r.max() / grid_size))
    n_a = 36
    gr = np.clip((r / grid_size).astype(int), 0, n_r - 1)
    ga = np.clip(((az + np.pi) / (2*np.pi) * n_a).astype(int), 0, n_a - 1)
    ground = np.zeros(len(pts_xyz), dtype=bool)
    for ri in range(n_r):
        for ai in range(n_a):
            idx = np.where((gr == ri) & (ga == ai))[0]
            if len(idx) < 3: continue
            z_min = pts_xyz[idx, 2].min()
            ground[idx] = pts_xyz[idx, 2] < (z_min + thresh)
    return np.where(ground)[0], np.where(~ground)[0]

# ------- 步骤2: DBSCAN聚类 -------
def cluster_points(pts, eps=0.5, min_pts=10):
    labels = DBSCAN(eps=eps, min_samples=min_pts).fit(pts).labels_
    return [np.where(labels == l)[0] for l in set(labels) if l != -1]

# ------- 步骤3: PCA-OBB拟合 -------
def fit_obb(points):
    PCA拟合有向包围盒, 返回(center, size)
    c = points.mean(axis=0)
    cov = np.cov((points - c).T)
    evals, evecs = np.linalg.eigh(cov)
    size = np.array([(points - c) @ evecs[:,i] for i in range(3)]).ptp(axis=1)
    return c, size[np.argsort(size)[::-1]]  # 最长轴在前

# ------- 步骤4: 雷达-激光关联 (匈牙利算法) -------
def associate_radar(radar_xy, obs_centers, max_dist=2.0):
    nr, no = len(radar_xy), len(obs_centers)
    cost = np.full((nr, no), 1e9)
    for i, rp in enumerate(radar_xy):
        for j, oc in enumerate(obs_centers):
            d = np.linalg.norm(rp - oc[:2])
            if d < max_dist: cost[i,j] = d
    return list(zip(*linear_sum_assignment(cost))))

# ------- 步骤5: 置信度融合 -------
def fuse_conf(li_conf, ra_conf=0, vi_conf=0, w=(0.5,0.3,0.2)):
    fused = li_conf * w[0]
    ws = w[0]
    if ra_conf > 0: fused += ra_conf * w[1]; ws += w[1]
    if vi_conf > 0: fused += vi_conf * w[2]; ws += w[2]
    return min(fused / ws, 1.0)

# ------- 主流程 -------
def run_fusion(lidar_pts, radar_targets, eps=0.5, min_pts=10):
    _, ng_idx = segment_ground(lidar_pts[:,:3])
    ng = lidar_pts[ng_idx][:,:3]
    clusters = cluster_points(ng, eps, min_pts)
    obstacles = []
    for i, cl in enumerate(clusters):
        if len(cl) < 5: continue
        c, s = fit_obb(ng[cl])
        if s.min() < 0.3: continue
        conf = min(len(cl)/200, 0.95)
        obstacles.append({'id':i, 'center':c, 'size':s, 'conf':conf, 'vel':np.zeros(3)})
    # 雷达关联 -> 补充速度
    if radar_targets:
        matches = associate_radar(
            np.array([[rt['x'], rt['y']] for rt in radar_targets]),
            [obs['center'] for obs in obstacles])
        for ri, oi in matches:
            obstacles[oi]['vel'] = radar_targets[ri]['vel_enu']
    for obs in obstacles:
        obs['conf'] = fuse_conf(obs['conf'], 0.7)
    return obstacles
from collections import defaultdict

def build_4d_grid(trajectories, res=1.0, horizon_s=10.0,
                  safety_xy=5.0, safety_z=3.0, sigma_grow=0.02):
    
    trajectories: {drone_id: [(t_ms, x, y, z, vx, vy, vz, sigma), ...]}
    返回: grid = {(ix,iy,iz,it): {drone_id, ...}}
    
    grid = defaultdict(set)
    t_step = 100  # ms
    h_steps = int(horizon_s * 1000 / t_step)

    for did, states in trajectories.items():
        for si in range(len(states)):
            end = min(si + h_steps, len(states))
            for fi in range(si, end):
                t, x, y, z, vx, vy, vz, sigma = states[fi]
                dt = (fi - si) * 0.1  # s
                sigma_t = sigma + sigma_grow * dt  # 不确定性膨胀
                half_xy = safety_xy/2 + 3*sigma_t
                half_z  = safety_z/2  + 3*sigma_t

                ix0 = int((x - half_xy) / res)
                ix1 = int((x + half_xy) / res)
                iy0 = int((y - half_xy) / res)
                iy1 = int((y + half_xy) / res)
                iz0 = int((z - half_z)  / res)
                iz1 = int((z + half_z)  / res)
                it  = int(t / t_step)

                for ix in range(ix0, ix1+1):
                    for iy in range(iy0, iy1+1):
                        for iz in range(iz0, iz1+1):
                            grid[(ix, iy, iz, it)].add(did)
    return grid

def detect_conflicts(grid, drone_ids, safety_xy=5.0, safety_z=3.0):
    遍历全部机对, 找体素交集 → 冲突判定
    conflicts = []
    for i in range(len(drone_ids)):
        for j in range(i+1, len(drone_ids)):
            a, b = drone_ids[i], drone_ids[j]
            vox_a = {k for k,v in grid.items() if a in v}
            vox_b = {k for k,v in grid.items() if b in v}
            common = vox_a & vox_b
            if not common: continue

            # 按时间分组, 找最接近距离
            time_groups = defaultdict(list)
            for (ix,iy,iz,it) in common:
                time_groups[it].append((ix,iy,iz))

            for it, voxels in time_groups.items():
                # 体素中心距离近似 (两机在同一体素 → 距离<体素对角线)
                dist_est = 1.73  # √3 for 1m grid
                sep = min(safety_xy, safety_z)
                level = 2 if dist_est < sep else 1
                conflicts.append({
                    'time_ms': it * 100, 'drone_a': a, 'drone_b': b,
                    'closest_m': dist_est, 'margin_m': sep, 'level': level
                })
    return conflicts

        'len_min': lengths[0], 'len_max': lengths[-1],
        'len_median': lengths[n//2],
        'est_hb_ms': int(np.median(intervals)) if intervals else 1000,
    }

# ------- 模板匹配 -------
def match_template(feat):
    best_name, best_score = 'Unknown', 0.0
    for tmpl in TEMPLATES:
        score = 0.0
        if any(h in dict(feat['top_headers']) for h in tmpl['headers']):
            score += 0.4
        if tmpl['len_range'][0] <= feat['len_median'] <= tmpl['len_range'][1]:
            score += 0.3
        if feat['est_hb_ms'] > 0:
            err = abs(feat['est_hb_ms'] - tmpl['hb_ms']) / tmpl['hb_ms']
            if err < 0.2:   score += 0.2
            elif err < 0.5: score += 0.1
        if score > best_score:
            best_score, best_name = score, tmpl['name']
    return best_name, min(best_score, 0.95) if best_score >= 0.5 else ('Unknown', best_score)

# ------- 注册状态机 -------
class RegistryStateMachine:
    STATES = ['DISCOVERED','IDENTIFYING','CAP_REQ','CAP_RECV','REGISTERED','FAILED']
    def __init__(self, protocol, confidence):
        self.state = 'DISCOVERED'
        self.protocol = protocol
        self.confidence = confidence
        self.capability = {}
        self.retries = 0
        self.log = []

    def step(self, new_state, detail=''):
        self.log.append(f'[{self.state}] → [{new_state}] {detail}')
        self.state = new_state

    def run(self, allocator, drone_type=3):
        self.step('IDENTIFYING', f'proto={self.protocol} conf={self.confidence:.0%}')
        if self.protocol == 'Unknown':
            self.step('FAILED', 'unknown protocol'); return None
        self.step('CAP_REQ')
        # 模拟能力查询 (实际通过通信链路上报)
        self.capability = {
            'max_speed': 15, 'max_altitude': 500, 'max_endurance_s': 1800,
            'has_lidar': 1, 'supports_vel_ctrl': 1, 'supports_mission': 1
        }
        self.step('CAP_RECV', f'{len(self.capability)} fields')
        num_id = allocator.allocate(drone_type)
        self.step('REGISTERED', f'id={num_id}')
        return num_id

# ------- 编号分配器 -------
class IDAllocator:
    PREFIX = {0:'R', 1:'D', 2:'L', 3:'U'}
    def __init__(self): self._cnt = {k:0 for k in self.PREFIX}
    def allocate(self, dtype=3):
        self._cnt[dtype] += 1
        return f'UAV_{self.PREFIX[dtype]}{self._cnt[dtype]:02d}'

import matplotlib.pyplot as plt

TDMA_SLOTS = 36        # 每帧时隙数
TDMA_FRAME_MS = 30     # 帧周期
SLOT_KBPS = 5.6        # 单时隙容量 (kbps, GFSK 200kbps, 833us)

def greedy_allocate(demands):
    
    demands: [{'id':nid, 'dir':'UL'/'DL', 'rate_kbps':r, 'priority':p}, ...]
    返回: slot_table = [{'node':nid, 'dir':d, 'pri':p}, ...] × 36
    
    queue = sorted(demands, key=lambda d: (d['priority'], -d['rate_kbps']))
    slots = [None] * TDMA_SLOTS
    slots[0] = slots[-1] = {'node': -1, 'dir': 'GUARD', 'pri': 0}  # 保护时隙

    alloc_cnt = defaultdict(int)
    for dm in queue:
        needed = max(1, int(np.ceil(dm['rate_kbps'] / SLOT_KBPS)))
        for s in range(1, TDMA_SLOTS - 1):
            if alloc_cnt[dm['id']] >= needed: break
            if slots[s] is None:
                slots[s] = {'node': dm['id'], 'dir': dm['dir'], 'pri': dm['priority']}
                alloc_cnt[dm['id']] += 1
    return slots

def estimate_delays(slot_table, node_demands):
    估算每节点的平均/最大时隙等待延迟 (ms)
    delays = {}
    for dm in node_demands:
        nid = dm['id']
        # 找到该节点的时隙位置
        positions = [i for i, s in enumerate(slot_table) if s and s['node'] == nid]
        if not positions: continue
        # 时隙间隙 → 等待时间 (帧内均匀分布假设)
        gaps = np.diff(positions)
        avg_gap = np.mean(gaps) if len(gaps) > 0 else TDMA_SLOTS
        delays[nid] = {
            'avg_ms': avg_gap * TDMA_FRAME_MS / TDMA_SLOTS / 2,  # 期望等待=间隙/2
            'max_ms': max(gaps) * TDMA_FRAME_MS / TDMA_SLOTS if len(gaps) > 0 else TDMA_FRAME_MS
        }
    return delays

def plot_gantt(slot_table, output='tdma_gantt.png'):
    绘制TDMA帧甘特图
    fig, ax = plt.subplots(figsize=(14, 4))
    colors = {None: 'lightgray', 'GUARD': 'darkgray',
              'UL': '#4CAF50', 'DL': '#2196F3'}
    for i, s in enumerate(slot_table):
        if s is None:
            c, label = 'lightgray', ''

    ax.set_xlabel('Slot index'); ax.set_title('TDMA Frame Slot Allocation (Gantt)')

import heapq
from matplotlib.patches import Polygon

# ------- 栅格路径搜索 -------
def astar_grid(start, goal, grid_2d, allow_diag=True):
    
    2D栅格搜索。
    grid_2d: 0=自由, 1=障碍
    start/goal: (row, col) 栅格索引
    返回: [(row,col), ...] 路径点列表 或 None (无解)
    
    rows, cols = grid_2d.shape
    neighbors = [(0,1),(0,-1),(1,0),(-1,0)]
    if allow_diag:
        neighbors += [(1,1),(1,-1),(-1,1),(-1,-1)]

    def h(a, b): return np.sqrt((a[0]-b[0])**2 + (a[1]-b[1])**2)  # 欧氏距离启发式

    open_set = [(h(start, goal), 0, start)]
    came_from = {}
    g_score = {start: 0}

    while open_set:
        _, _, current = heapq.heappop(open_set)
        if current == goal:
            # 回溯路径
            path = [current]
            while current in came_from:
                current = came_from[current]
                path.append(current)
            return path[::-1]

        for dr, dc in neighbors:
            nr, nc = current[0]+dr, current[1]+dc
            if not (0 <= nr < rows and 0 <= nc < cols): continue
            if grid_2d[nr, nc] == 1: continue  # 障碍
            move_cost = np.sqrt(dr**2 + dc**2)
            tent_g = g_score[current] + move_cost
            if tent_g < g_score.get((nr, nc), float('inf')):
                came_from[(nr, nc)] = current
                g_score[(nr, nc)] = tent_g
                f = tent_g + h((nr, nc), goal)
                heapq.heappush(open_set, (f, id((nr, nc)), (nr, nc)))
    return None  # 无可行路径

# ------- 空域冲突检测 -------
def check_airspace_conflict(new_zone, existing_zones):
    
    new_zone: {'type':'no_fly'/'corridor'/'return', 'polygon':[(x,y),...]}
    existing_zones: 已有的空域列表
    返回: 冲突的空域列表
    
    from shapely.geometry import Polygon as SPoly
    new_poly = SPoly(new_zone['polygon'])
    conflicts = []
    for ez in existing_zones:
        if new_poly.intersects(SPoly(ez['polygon'])):
            conflicts.append(ez)
    return conflicts

# ------- 航线重规划主流程 -------
def replan_routes(drones, airspace_changes, grid_res=1.0, bounds=(1000,1000)):
    
    drones: {drone_id: {'plan':[(x,y),...], 'pos':(x,y)}}
    airspace_changes: 空域变更事件序列
    返回: 每架受影响无人机的重规划航线
    
    # 构建障碍栅格 (禁飞区=1, 自由=0)
    w, h = int(bounds[0]/grid_res), int(bounds[1]/grid_res)
    grid = np.zeros((h, w), dtype=np.uint8)

    for change in airspace_changes:
        if change['type'] == 'no_fly_add':
            # 将禁飞区多边形栅格化
            poly = np.array(change['polygon']) / grid_res
            # ... 栅格化填充 (省略具体实现)
            pass

    results = {}
    for did, drone in drones.items():
        # 检查原航线是否穿越禁飞区
        needs_replan = False
        for wp in drone['plan']:
            r, c = int(wp[1]/grid_res), int(wp[0]/grid_res)
            if 0 <= r < h and 0 <= c < w and grid[r, c] == 1:
                needs_replan = True; break

        if not needs_replan: continue

        start = (int(drone['pos'][1]/grid_res), int(drone['pos'][0]/grid_res))
        goal  = (int(drone['plan'][-1][1]/grid_res), int(drone['plan'][-1][0]/grid_res))
        new_path = astar_grid(start, goal, grid)
        if new_path:
            results[did] = [(c*grid_res, r*grid_res) for (r,c) in new_path]
        else:
            results[did] = None  # 无可行路径 → 需要人工干预
    return results
